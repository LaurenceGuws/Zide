# Keyboard Alternate Metadata Contract (PA-05a)

Date: 2026-02-23
Status: draft (design contract for future implementation)
Owner: agent

## Purpose

Define the minimum input-event metadata required to implement layout-aware kitty
keyboard `report_alternate_key` parity correctly (beyond the current US-ASCII
shifted-char heuristic in the legacy char path).

## Why This Is Needed

The current legacy terminal input path encodes character input from:
- resolved Unicode codepoint (`char`)
- modifier bits (`shift`, `alt`, `ctrl`, ...)

That is insufficient for correct kitty alternate-key reporting on non-US layouts
because the encoder may need to distinguish:
- physical key identity vs produced text
- base/unshifted layout codepoint vs shifted codepoint
- layout alternate(s) used for shortcut matching

Without this metadata, additional heuristics will produce incorrect
`key:shifted[:alternate]` fields for some layouts.

## Required Metadata (Minimum)

Proposed per key event fields (names are indicative):

- `physical_key`:
  - stable logical/physical key identity (e.g. key enum/scancode-like)
  - needed to correlate the pressed key independent of produced text

- `produced_text_utf8`:
  - UTF-8 text generated by the event (may be empty)
  - supports kitty associated text / alternate inference when text is present

- `base_codepoint`:
  - unshifted layout codepoint for the same key, if known
  - example: pressing `Shift+;` on US yields text `:` while base codepoint is `;`

- `shifted_codepoint`:
  - shifted layout codepoint for the same key, if known and distinct
  - example: US `a` key with shift -> `A`

- `alternate_layout_codepoint` (optional):
  - additional layout-dependent alternate codepoint when it differs from both
    main and shifted values
  - required for full `key:shifted:alternate` reporting parity

- `text_is_composed` (optional but recommended):
  - indicates IME/compose result where base/shifted inference should be treated
    carefully or suppressed

## Encoder Expectations

With the metadata above, kitty `report_alternate_key` encoding can follow:
- only apply alternates when the event is already encoded as CSI-u (progressive enhancement rule)
- emit `key:shifted` when shifted alternate is known and distinct
- emit `key:shifted:alternate` when layout alternate is known and distinct
- avoid fabricating alternates when metadata is absent/ambiguous

## Non-Goals (This Step)

- No runtime behavior change
- No input-event type changes yet
- No non-US fixtures yet

## SDL / Platform Feasibility Notes (PA-05b research)

Current data available in our SDL input pipeline:
- physical key identity via SDL scancode (`event.keysym.scancode`)
- layout-translated key symbol (`event.keysym.sym`)
- modifier state
- text input UTF-8 bytes from `SDL_EVENT_TEXT_INPUT`
- composition/editing state from `SDL_EVENT_TEXT_EDITING`

What this enables now:
- stable `physical_key` (from scancode)
- `produced_text_utf8` and `text_is_composed`
- a better future path to infer `base_codepoint` / `shifted_codepoint` in some cases
  using SDL scancode + modifier translation

Important SDL limits / caveats:
- `keysym.sym` is a single layout-translated keycode for the event, not a full
  tuple of base/shifted/alternate layout outputs.
- `SDL_EVENT_TEXT_INPUT` provides committed text (possibly IME/composed), which
  is too late / too lossy to reconstruct the physical-key layout mapping in all
  cases.
- IME / dead-key composition can produce text that should not imply alternate
  key fields (hence `text_is_composed` suppression in the encoder).
- SDL keyboard translation behavior can be affected by keycode options hints;
  for reliable layout translation from scancode+modstate, SDL docs recommend
  explicit `SDL_GetKeyFromScancode(...)` calls instead of assuming event keycode
  semantics.

Feasibility conclusion:
- Partial layout-aware alternate support is feasible with current plumbing.
- Full parity (`key:shifted:alternate` across layouts/IME paths) still likely
  needs explicit scancode+modstate translation calls (new wrapper in
  `src/platform/sdl_api.zig`) and careful platform/IME behavior testing.
